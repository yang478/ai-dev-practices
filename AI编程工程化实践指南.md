# AI 编程工程化实践指南

> 来源：整理自「AI编程工程化」系列视频（林湛星）的核心观点。  
> 适用对象：个人开发者、小团队；不适用未经验调整的大规模团队协作。

## 一、核心结论

AI 编程的最大风险不是代码写得太烂，而是 AI **一开始就把需求理解错了**，并在后续非常自信地把错误的东西做完整。

真正拉开差距的问题：**项目能不能持续迭代下去？**

AI 时代下，开发者的价值从「写代码」转移到「做判断」：
- 判断哪些决定值得保留
- 判断哪些边界必须锁住
- 判断 AI 的输出是否偏离意图

## 二、两个核心概念

### 1. Spec（规格 / 决策文档）

Spec 不是普通文档，而是 **"已经想清楚、以后不再第二次讨论的决定"**。

它回答：
- 这个功能是什么？
- 为什么这样设计？
- 边界在哪里？
- 禁止做什么？

Spec 的本质作用是 **帮你自己止损**。它把你那个会不停摇摆的判断力固定下来，让每次新开的 AI 对话不用再重新争论同一个问题。

#### Spec 的分类

| 类型 | 作用 | 示例 |
|------|------|------|
| 自然语言决策类 Spec | 让 AI 理解业务意图和边界 | "宠物饱腹度 ≥ 90 时拒绝喂食" |
| Harness 工程类 Spec | 把核心业务逻辑外包给自动化检查 | 喂养接口的输入输出断言 |

自然语言 Spec 是「软约束」：AI 可能遗忘或误解。  
Harness Spec 是「硬约束」：用可执行测试强制守住底线。

### 2. Harness（自动化测试 / 门禁）

Harness 不是传统意义上的"抓 bug 测试"，它的核心作用是 **把信任外包出去**。

个人开发者面对的真实困境：
- 所有代码都是 AI 写的
- 所有审查都要自己一个人完成
- AI 最危险的是写出「看起来对、跑起来不报错、但悄悄违背核心决定」的代码

Harness 让你敢对 AI 的改动说"行"，而不必逐行细读。一旦 AI 触碰边界，Harness 自动上报。

#### Harness 的目标不是覆盖率

公司 KPI 式的覆盖率对个人开发者没有意义。Harness 只需要精准覆盖：
- 已经确定不会再变的核心业务
- 一旦破坏会导致连锁反应的地方
- 历史反复踩坑的 bug
- 出错后难排查的业务流程

## 三、项目生命周期两阶段

### 阶段一：0 → 1（MVP 阶段）—— 放开抡

**原则：不要过早写 Spec。**

过早写大量 Spec 是一种「伪装成严谨的拖延」。你会给一栋还没决定要不要盖的楼提前搭满脚手架。

MVP 阶段真正该做的：
1. **写好项目宪章（Agent 文档）**
   - 产品是什么？
   - 产品不是什么？（反向边界比正向更重要）
   - 你希望 AI 以什么风格协助你？
   - 建议风格：有因果链（是什么、为什么、怎么做），先理解再动手
2. **与 AI 充分讨论需求**
   - 不要直接让 AI 写代码
   - 让 AI 先产出 MVP 阶段的项目文档，验证它理解的是否和你脑中一致
   - 意图对齐越早，成本越低
3. **让 AI 自主完成 MVP**
   - 代码质量要求接近零：只要实现效果、能通过真实使用验证即可
   - 所有问题一次性整理后，新开对话反馈给 AI 修复
   - 修复计划是临时产物，不是资产
4. **MVP 跑通后，做第一次"编辑"**
   - 审阅 AI 在 MVP 中自发做出的所有决定
   - 给它们三个命运：固化成 Spec / 重构掉 / 先放着继续演化

**MVP 阶段只写一样东西：项目宪章。**

### 阶段二：1 → 10（持续迭代阶段）—— 建立秩序

**原则：地基打好后，逐步引入 Spec 和 Harness。**

#### 打地基：重构 MVP

在 MVP 验收通过后，新开对话让 AI 自行审查架构，要求：
- 模块化
- 功能与主架构解耦
- 高内聚、低耦合

重构结束并验收通过后，地基才算打好。

#### 目录与文档规范

- 项目根目录保持干净
- 建立专门的 `docs/` 目录管理文档
- 根目录的 `AGENTS.md` 或项目宪章中写清文档索引
- AI 按需读取，而不是一次性加载全部

#### 沉淀第一批 Spec

地基打好后，先写两类确定不会再变的决定：
1. **项目方向类**：产品是什么、核心定位
2. **功能边界类**：明确不做什么

把这些记录为 ADR（Architecture Decision Record），按顺序编号。

#### 新功能开发流程

1. **新开对话**，排除上下文噪声
2. 描述新需求，多轮讨论直到 AI 真正理解
3. 让 AI 直接落地（不需要再新开对话）
4. **验收**：自己真实操作，记录问题
5. 一次性整理问题，新开对话让 AI 出修复计划
6. 修复后，让 AI 回头审查：**哪些约束值得沉淀为 Spec？**
7. 自己筛选，把真正的原则写成 Spec

#### 何时引入 Harness

当某块业务逻辑已经确定，且你 **不允许 AI 以后偷偷改动它** 时，就把它放进 Harness。

典型场景：
- 喂养宠物的核心业务逻辑
- 登录、支付、权限、数据保存等高风险流程
- 历史反复踩坑的地方
- 出错后牵连多、难排查的地方

Harness 可以从最简单的脚本开始：
- 准备测试条件
- 执行检查
- 输出通过 / 失败

## 四、AI 的两大特性必须时刻记住

### 1. 上下文不是数据库

上下文窗口像一只水桶：
- 水桶容量有限
- 水装满后会「熬汤」（压缩）
- 熬汤后细节丢失，只剩模糊印象
- 模型不会主动告诉你"我不确定了"

**窗口再大也只是把熬汤的时刻推迟。**

应对方法：
- 不要把所有文档塞进一份文件
- 文档按需拆分、按需加载
- 项目宪章每次新对话都加载
- 模块文档、ADR、Harness 只在需要时加载

### 2. AI 是概率模型，不是规则引擎

- 文档 AI "看到了"，不等于它"会稳定照做"
- 为了当前功能跑通，AI 可能顺手改掉原有正常逻辑
- 文档只能降低错误概率，不能降到零

**必须用 Harness 形成闭环：文档定边界，测试验结果。**

## 五、常见误区

| 误区 | 正确理解 |
|------|----------|
| 一开始就写完整 Spec | MVP 阶段只写项目宪章，Spec 滞后于代码 |
| Spec 越多越好 | Spec 是稳定决定的"户口"，没稳定的决定不配写 |
| 有 Spec 就要有 Harness | 不是一一对应，只给核心、不变、高风险业务加 Harness |
| 文档越多 AI 越听话 | 文档过多会淹没关键信息，反而造成上下文污染 |
| Harness 就是普通测试 | Harness 是「考场监考 + 判卷系统」，目标是守住边界，不是覆盖率 |
| AI 能自我验收 | 验收必须发生在 AI 之外，AI 的自我判断不能当结果 |

## 六、行动清单

### MVP 阶段
- [ ] 写好项目宪章：是什么、不是什么、AI 协作风格
- [ ] 与 AI 充分讨论需求，产出 MVP 项目文档
- [ ] 验证 AI 理解正确后，让它自主开发 MVP
- [ ] 自己真实使用 MVP，记录问题
- [ ] 一次性反馈问题，新开对话修复
- [ ] MVP 跑通后，决定哪些决定值得固化

### 迭代阶段
- [ ] 重构 MVP，做到模块化、高内聚低耦合
- [ ] 建立 `docs/` 文档目录和索引
- [ ] 写下项目方向类 ADR 和功能边界类 ADR
- [ ] 每个新功能：新开对话 → 对齐需求 → 开发 → 验收 → 沉淀 Spec
- [ ] 对核心、不变、高风险业务逐步建设 Harness
- [ ] 每次修复后区分：一次性 bug / 值得沉淀的原则

### 日常原则
- [ ] 每次纠正 AI 一个原则性问题时，问自己：这件事还会再发生吗？
- [ ] 如果会，写成 Spec
- [ ] 如果涉及核心业务，再加 Harness
- [ ] 文档按需加载，避免一次性塞满上下文
- [ ] 验收永远发生在 AI 之外

## 七、一句话总结

> **AI 负责探索和起草，你负责判断和拍板；Spec 把决定固定下来，Harness 把边界执行下去。**
